Converting Data from Relational

To bring relational data into a MarkLogic database, it is important to understand the difference between modeling data in the relational world versus modeling data in the NoSQL world.

Relational databases model data so that the content for a document (e.g. an order document) is often chopped up and spread across multiple tables (i.e. 'normalized'), whereas most NoSQL databases model the content for a document as a document (i.e. 'denormalized').

Techniques for Converting Relational Data

In a perfect scenario, the original documents (before they went into a relational database) are still available, and then loaded into MarkLogic.

If the original documents are not available, but the data is in a relational database, the next best option is to reconstitute the documents from the relational database (i.e. denormalize the data).

To reconstitute documents from a relational database, a couple of techniques can be used, including running a SQL query against the relational database, to denormalize the documents, and then import the documents into MarkLogic, or to export the relational tables as .csv files and then, in MarkLogic, stitch the rows together, to reconstitute the documents.

Apache NiFi can make the process of converting relational data relatively easy, running a SQL query against a relational database, and then using the results to create and load the resulting documents into MarkLogic. For a 30-minute hands-on video that shows how to use Apache NiFi to bring relational data into MarkLogic, visit developer.marklogic.com/video/data-modeling.
